## 1. 서론 (Introduction)

본 프로젝트는 MIDI 데이터를 기반으로 인간의 해부학적 특성과 음악적 해석이 결합된 고정밀 피아노 연주 시뮬레이션을 생성하는 것을 목표로 합니다. 특히 `01_Fingering` 모듈은 단순한 시각화를 넘어, 언리얼 엔진(UE5) 등의 3D 환경에서 실제 물리적 IK(Inverse Kinematics)를 구동하기 위한 **'애니메이션 가이드 데이터'**를 생성하는 핵심 브레인 역할을 수행합니다.

## 2. 엔진의 진화 과정 (Version History)

### V1 & V2: 기초 동적 프로그래밍 (The Foundation)
*   **개념:** 각 음표를 독립적인 노드로 보고 가장 편한 손가락을 찾는 기초 DP 구현.
*   **한계:** 화음(Chord) 처리가 미흡하여 동시 타건 시 손가락이 꼬이는 논리적 오류 발생.

### V4: 해부학적 가동 범위(ROM)의 도입 (Biomechanical Constraints)
*   **혁신:** 인체의 물리적 한계를 수학적 상수로 도입 (`MAX_SPAN`).
*   **주요 기능:**
    *   손가락 사이의 거리(Semitones)를 계산하여 물리적으로 불가능한 스트레칭에 막대한 페널티 부여.
    *   '엄지 밑으로 넣기(Thumb-under)' 외의 비정상적인 손가락 교차 차단.
    *   손목의 Yaw(좌우), Roll(기울기) 각도 산출 로직 추가.

### V5: 폴리포니 및 성부 분석 엔진 (Polyphonic Voice Leading) - **현재**
*   **혁신:** 음악적 맥락(Musical Context) 파싱 도입.
*   **주요 기능:** 한 손 내에서도 멜로디와 반주를 구분하여, 멜로디 라인의 수평적 연결성(Legato)을 최우선으로 고려하는 운지법 생성.

## 3. 핵심 기술 상세 (Technical Deep Dive)

### 3.1 Chord-based DP Solver
단일 음표가 아닌 **'화음 그룹'**을 하나의 상태(State)로 처리합니다.
*   **화음 그룹화**: 30ms 이내에 시작되는 음표들을 하나의 화음으로 묶어 처리함으로써 연산 복잡도를 줄이고 연주의 일관성을 확보합니다.
*   **비용 함수 (Cost Function)**:
    $$TotalCost = \sum (WristMove) + \sum (FingerDifficulty) + \sum (AnatomicalPenalty) + \sum (RolePenalty)$$
    *   **WristMove**: 이전 포지션에서 현재 포지션까지의 손목 이동 거리.
    *   **FingerDifficulty**: 4, 5번 손가락의 태생적 약함을 반영한 비용.
    *   **AnatomicalPenalty**: 블랙키 연주 시 엄지 사용, 과도한 스팬 등에 대한 페널티.

### 3.2 성부 분리 및 음악적 휴리스틱 (Musical Heuristics)
엔진은 MIDI 데이터를 파싱할 때 각 음표에 `Role`을 부여합니다.
*   **MELODY (Red)**: 가장 높은 음역대. 선율의 매끄러움을 위해 4, 5번 손가락 사용 권장 및 1번(엄지) 사용 자제 (선율의 끊김 방지).
*   **BASS (Blue)**: 가장 낮은 음역대. 안정적인 지지를 위해 5번(새끼) 또는 1번(엄지) 선호.
*   **INNER (Yellow)**: 화음을 채우는 역할. 2, 3, 4번 손가락을 효율적으로 배정.

### 3.3 해부학적 물리 모델링
*   **Wrist Yaw**: 손가락들이 누르는 건반의 평균 위치와 손목의 중심축 사이의 오프셋을 계산하여 `-35.0° ~ +35.0°` 범위 내의 각도를 산출합니다.
*   **Wrist Roll**: 엄지가 흑건을 누르거나 새끼손가락이 흑건을 누를 때 발생하는 손목의 기울어짐을 모사합니다 (±20.0°).

## 4. 중대 결함 해결: 왼손 역전 문제 (Case Study)

### [현상] 손가락 X-자 꼬임
시각화 결과, 왼손의 엄지(1번)가 가장 낮은 피치를 누르려 하고 새끼(5번)가 높은 피치를 누르려 하면서 물리적으로 불가능한 X자 형태가 관찰되었습니다.

### [분석]
기존 엔진이 "낮은 피치 = 낮은 번호 손가락"이라는 오른손 중심의 논리를 양손에 동일하게 적용한 것이 원인이었습니다. 인체 구조상 왼손은 오른손의 거울상(Mirror)이므로 매핑 논리가 반전되어야 합니다.

### [해결]
1.  **조합 논리 반전**: 왼손 연산 시 손가락 번호 조합을 `(1, 2, 3)`이 아닌 `(5, 4, 3)` 등으로 역순 배정.
2.  **교차 판정 미러링**: 오른손은 상행 시 엄지가 밑으로(Thumb-under), 왼손은 하행 시 엄지가 밑으로 들어가는 동작이 정상임을 반영하여 교차 페널티 로직을 각 손의 방향성에 맞게 재구축.
3.  **결과**: 수정 후 왼손 운지가 X자 꼬임 없이 평행하고 자연스러운 동작을 유지함을 확인.

## 5. 실시간 시뮬레이터 시스템 (Simulation System)

### 5.1 Event-based MIDI Engine (`pygame.midi`)
기존의 파일 재생 방식의 한계를 극복하기 위해 실시간 이벤트 송출 방식을 채택했습니다.
*   **정밀도**: 프레임 단위(`FPS=30`)로 타임라인을 쪼개어, 해당 프레임에 진입하는 모든 `Note On` 이벤트를 시스템 MIDI 장치로 즉각 전송합니다.
*   **탐색(Seek)**: 사용자가 타임라인을 옮기면 즉시 `panic()`(모든 소리 끔) 명령을 내리고 새 위치의 이벤트를 즉시 재생하여 시각-청각적 일치감을 제공합니다.

### 5.2 지능형 시각화 (Multi-segment Coloring)
사용자의 직관적인 이해를 돕기 위해 손가락 마디별 이중 색상 시스템을 구현했습니다.
*   **안쪽 마디 (Musical Role)**: 현재 이 손가락이 어떤 음악적 성부(멜로디/베이스/내성)를 담당하는지 표시.
*   **끝 마디 (Finger Identity)**: 1~5번 손가락 중 정확히 어떤 손가락인지 고유 색상(빨/주/노/초/파)으로 표시.
*   **효과**: 복잡한 화음 연주 시에도 "엄지가 베이스를 치고 있는가, 아니면 내성을 보조하고 있는가?"를 즉각적으로 파악 가능.

## 6. 출력 데이터 사양 (JSON Specification)

최종 생성되는 `mario_polyphonic_result.json`은 다음과 같은 고밀도 정보를 포함합니다.
*   `pitch`: MIDI 음고 (21~108)
*   `start_ms / duration_ms`: 1/100ms 단위 정밀 시간
*   `hand`: 왼손/오른손 구분
*   `finger`: 최적화된 손가락 번호 (1~5)
*   `role`: 성부 역할 (MELODY, BASS, INNER)
*   `wrist_yaw_deg / wrist_roll_deg`: UE5 IK용 회전 각도 가이드
*   `pressure / key_depth`: 타건 강도 기반 애니메이션 가중치
